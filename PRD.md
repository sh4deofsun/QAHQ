# Product Requirements Document (PRD) - QAHQ

## 1. Introduction
**QAHQ** is a centralized, modular headquarters application designed for QA and Tester teams. It leverages **FastAPI** to provide a robust backend for managing, analyzing, and reporting on test results generated by **RobotFramework**.

## 2. Problem Statement
QA teams often struggle with:
- Decentralized test result management.
- Lack of immediate visibility into test execution statistics.
- Manual processing of RobotFramework XML outputs.
- Difficulty in integrating test reporting into automated pipelines.

## 3. Goals & Objectives
- **Centralization**: Provide a single point of entry for QA operations.
- **Distributed Architecture**: Enable a master-worker architecture where the HQ manages multiple remote workers.
- **Real-time Communication**: Use WebSockets for real-time communication between HQ and workers.
- **AI Integration**: Implement Model Context Protocol (MCP) to allow AI agents to query system state and test data.
- **Automation**: Automate the parsing and reporting of RobotFramework test results.
- **Modularity**: Allow easy addition of new features (Applications) without disrupting existing functionality.
- **Performance**: Ensure low-latency response times for API endpoints.

## 4. Functional Requirements

### 4.1 Core System
- **Framework**: Built on FastAPI for high performance and easy API documentation.
- **Communication Protocol**:
    - **REST API**: For external integrations and frontend communication.
    - **WebSockets**: For bi-directional, real-time communication between the HQ server and remote workers.
- **Middleware**: Custom middleware to track process time and sign responses (e.g., `X-Process-Time`, `Signature`).
- **Modular Architecture**: The system must support a plugin-like structure where new "Applications" can be added to the `Applications/` directory and easily integrated.

### 4.2 Distributed Worker System
- **Role**: Workers reside on different servers and execute tasks assigned by the HQ.
- **Connection**: Workers establish a persistent WebSocket connection to the HQ.
- **Heartbeat**: Implement heartbeat mechanism to monitor worker health.
- **Capabilities**: Workers must register their capabilities with HQ upon connection (e.g., `run_command`, `file_transfer`, `robot_execution`).
- **Task Distribution**: HQ dispatches tasks to specific workers based on registered capabilities and availability.

#### 4.2.1 Worker Features
- **Command Execution**: Ability to execute shell commands sent by HQ and stream output back.
- **File Transfer**:
    - **Receive**: Download files (e.g., test scripts, configurations) from HQ.
    - **Send**: Upload files (e.g., test reports, logs) to HQ.



### 4.2 Application Modules

#### 4.2.1 Test Counter (`TestCounterSingleton`)
- **Purpose**: Analyze RobotFramework output XML files to provide quick statistics.
- **Input**: RobotFramework `output.xml` file.
- **Output**: JSON response containing:
    - Total tests
    - Passed tests
    - Failed tests
    - Skipped tests
- **Endpoint**: `POST /get_test_count/`

#### 4.2.2 Test Report Sender (`TestReportSender`)
- **Purpose**: Process and send/store detailed test reports.
- **Input**: RobotFramework `output.xml` file.
- **Processing**: Uses `robot.rebot` to generate reports/logs from the uploaded XML.
- **Output**: JSON response with filenames of generated reports.
- **Endpoint**: `POST /send_test_report/`

### 4.3 Operations & Security
- **Logging**: Centralized logging configuration (in `Operations/`).
- **Security**: SSL configuration and other security measures (in `Securities/`).
- **Database**:
    - **Technology**: SQLite (default), extensible to other databases via SQLAlchemy.
    - **ORM**: SQLAlchemy must be used for all database interactions to ensure portability.
    - **Purpose**: Store latest test results, worker status, and execution history.
    - **Location**: `Databases/` directory (for SQLite file).

### 4.4 AI Integration (MCP)
- **Protocol**: Implement an MCP (Model Context Protocol) server.
- **Purpose**: Enable external AI agents to contextually understand and interact with the QAHQ system.
- **Exposed Resources/Tools**:
    - **Test Cases**: Ability to query available test suites, test cases, and their latest results.
    - **Worker Status**: Ability to check which workers are online, their capabilities, and current tasks.
    - **Logs**: Access to recent execution logs for debugging.


### 4.5 Authentication
- **Methods**:
    - **LDAP/LDAPS**: Primary method for general team members.
    - **Local Authentication**: Dedicated method for **Admin** users (credentials stored locally with secure hashing).
- **Mechanism**: Validate credentials against the appropriate source (LDAP server or Local DB).
- **Session**: Issue JWT or session tokens upon successful authentication.

## 5. Technical Requirements
- **Language**: Python 3.x
- **Web Framework**: FastAPI
- **Test Framework Integration**: `robotframework` (specifically `robot.api.ExecutionResult` and `rebot`).
- **Dependencies**: `pydantic`, `uvicorn` (implied), `requests`, `SQLAlchemy`.


## 6. Future Scope
- **Dashboard UI**: A frontend to visualize test trends over time.
- **Email Notifications**: Automated email alerts for failed tests (referenced in `Operations` folder description).


## 7. Assumptions & Constraints
- The system currently relies on uploading XML files; direct integration with CI/CD pipelines via other means (e.g., webhooks) is a potential enhancement.
- Local file storage is used for generated reports; cloud storage integration may be needed for scalability.
